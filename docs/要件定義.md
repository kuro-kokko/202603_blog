# 要件定義

> ステータス: 草案

---

## 1. 目的

Markdown ファイルを入力として静的な HTML サイトを生成するブログ用ジェネレーターを構築する。生成されたファイルは静的ホスティングサービスに配置するだけで動作し、サーバーサイドの実行環境を必要としない。

---

## 2. 利用シナリオ

| 役割 | 行動 |
|------|------|
| ブログ運営者 | `content/` に Markdown ファイルを追加・編集する |
| ブログ運営者 | `npm run build` を実行して静的ファイルを生成する |
| ブログ運営者 | 生成された `out/` を静的ホスティングにデプロイする |
| 読者 | ブラウザでブログ記事・タグ一覧・検索ページを閲覧する |

---

## 3. 機能要件

### 3.1 コンテンツ入力

| # | 要件 |
|---|------|
| F-01 | Markdown ファイル（`.md`）を記事コンテンツとして受け付ける |
| F-02 | YAML フロントマターで記事メタデータ（タイトル・日時・タグ等）を指定できる |
| F-03 | フロントマターの省略フィールドにはデフォルト値が補完される（詳細は §3.2） |
| F-04 | `[[slug]]` 記法（wiki リンク）でほかの記事へ内部リンクを張ることができる |
| F-05 | `[[slug|表示テキスト]]` でリンクテキストを上書きできる |
| F-06 | `draft: true` を指定した記事はビルド出力から除外される |
| F-07 | ファイル名が `_` で始まる記事はビルド出力から除外される（設定で変更可） |
| F-08 | `content/` ディレクトリを再帰的に走査し、サブディレクトリ構造を slug に反映する |

### 3.2 フロントマター仕様

| フィールド | 型 | 必須 | 省略時のデフォルト |
|------------|------|------|------------------|
| `title` | string | ○ | — |
| `createDateTime` | datetime | ○ | — |
| `updatedDateTime` | datetime | — | `createDateTime` と同値 |
| `tags` | string[] | — | `[]`（空配列） |
| `draft` | boolean | — | `false` |
| `description` | string | — | 本文冒頭 140 文字（Markdown 記法除去後） |
| `slug` | string | — | `relativePath` から `.md` を除いた値 |

### 3.3 ページ生成

| # | 要件 |
|---|------|
| F-10 | トップページ（記事一覧）を生成する |
| F-11 | 各記事の個別ページを生成する |
| F-12 | タグ別記事一覧ページを生成する |
| F-13 | 全タグの一覧ページを生成する |
| F-14 | 同一日付に複数の記事がある場合のみ、日付別一覧ページを生成する |
| F-15 | 検索ページを生成する |
| F-16 | 記事 URL は `/slug/` 形式（クリーン URL）とする |

### 3.4 メタデータ・インデックス

| # | 要件 |
|---|------|
| F-20 | タグごとに記事を集約した TagMap を構築する |
| F-21 | wiki リンクを解析し、被リンク記事に BacklinkMap を紐付ける |
| F-22 | 記事を月単位で集約した CalendarMap を構築する（設定で有効/無効） |
| F-23 | 全文検索用の JSON インデックス（`search-index.json`）を生成する |

### 3.5 UI コンポーネント

| # | 要件 |
|---|------|
| F-30 | 2 カラムレイアウト（メイン + サイドバー）でページを構成する |
| F-31 | サイドバーにプロフィールカード（名前・自己紹介・外部リンク）を表示できる |
| F-32 | 記事ページのサイドバーに目次（h2/h3）を自動生成・表示する |
| F-33 | トップページのサイドバーに月次カレンダーを表示できる（設定で有効/無効） |
| F-34 | カレンダーで記事のある日付をリンク付きでハイライトする |
| F-35 | 同日複数記事の場合、カレンダーの日付リンクは日付別一覧ページへ遷移する |
| F-36 | 記事ページにバックリンク（この記事へのリンク）一覧を表示する |
| F-37 | バックリンクの表示位置（本文直下 / サイドバー）を設定で切り替えられる |
| F-38 | タグ一覧でタグ名と記事件数を表示し、タグ別ページへリンクする |
| F-39 | 記事一覧は `createDateTime` の降順に並べる |

### 3.6 検索

| # | 要件 |
|---|------|
| F-40 | クライアントサイドで全文検索を行う（サーバー不要） |
| F-41 | 日本語（CJK）テキストを検索できる（ユニグラム＋バイグラムトークナイズ） |
| F-42 | タイトル・説明文・タグを検索対象フィールドとする |
| F-43 | タイトルへのマッチを説明文・タグより高スコアで評価する |
| F-44 | 入力から 200 ms のデバウンス後に検索を実行する |
| F-45 | URL クエリパラメータ `?q=` による初期検索状態の復元に対応する |

### 3.7 セキュリティ

| # | 要件 |
|---|------|
| F-50 | Markdown 本文中の `<script>` タグはサニタイズして除去する |
| F-51 | `onerror` 等のイベントハンドラ属性はサニタイズして除去する |
| F-52 | `javascript:` スキームの `href` 属性はサニタイズして除去する |
| F-53 | 検索結果の表示はテキストエスケープを行い XSS を防ぐ |

---

## 4. 非機能要件

### 4.1 互換性
- Node.js 20 以上で動作すること
- 生成された HTML はモダンブラウザ（Chrome/Firefox/Safari/Edge 最新版）で正常表示されること

### 4.2 パフォーマンス
- ビルド時間: 記事 100 件以下のサイトで 30 秒以内に完了すること
- 検索インデックスはオンデマンドで非同期ロードし、初回ページロードをブロックしないこと

### 4.3 保守性
- TypeScript により型安全なコードベースを維持すること
- ページコンポーネントは Preact の JSX で記述し、UI ロジックと HTML 生成を分離すること

### 4.4 デプロイ
- 出力は静的ファイルのみで構成し、サーバーサイド実行環境を不要とすること
- GitHub Pages・Netlify・Vercel 等の静的ホスティングに追加設定なしで配置できること

---

## 5. 制約・前提条件

- コンテンツは `content/` ディレクトリ配下の Markdown ファイルとする
- スタイリングは `assets/style.css` で一元管理する
- クライアント JavaScript は検索機能のみとし、ページ本体の表示は HTML/CSS のみで完結させる
- wiki リンクの存在チェック（リンク切れ検出）はビルド時に行わない
